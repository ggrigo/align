# Archive Format — Datapoints, Manifest, Files

This reference defines the on-disk format /align uses to durably store correction sessions. The archive is the input to `/retro` and the long-term asset that accumulates calibrated user taste.

## The datapoint concept

Each /align session produces a **datapoint**: one `.html` file (the form, with the claim array embedded) plus one `.md` file (the user's ratings and notes). The pair is one row in the manifest and one folder entry in the archive.

Datapoints are immutable once archived. If a correction was wrong, the next session catches it as a new claim about prior state, not by editing the archive.

## File locations

> **Note on `rhythm/` prefix.** Paths in this doc default to `rhythm/align-archive/` because /align's first producer was /rhythm. When running /align in a different working folder (e.g., `clients/adidas/`), the archive can live under that folder instead — `clients/adidas/align-archive/`, `clients/adidas/align-index.md`, etc. Substitute the prefix accordingly.

| What | Where | Purpose |
|------|-------|---------|
| Active `.html` | The active working folder (e.g., `rhythm/`, `clients/adidas/...`) | Local context; the file the user opens to rate claims |
| Archived `.html` | `rhythm/align-archive/` | Permanent record; `/retro` reads this |
| Archived `.md` | `rhythm/align-archive/` | Permanent record; the parsed corrections live here |
| Manifest | `rhythm/align-index.md` | Quick lookup, one row per session, stats over time |
| Pending corrections queue | `rhythm/align-archive/align-corrections-pending.md` | Smart-memory writes that couldn't land because the tool wasn't wired. Drained by the next session that has the tool. See `references/integrations.md` |
| Browser dropbox | `incoming-docs/` (Cowork) or `~/Downloads/` (Claude Code) | Where the user's downloaded `.md` lands before /align picks it up |

The archive folder defaults to `rhythm/align-archive/` (with the same substitution caveat above for non-rhythm working folders). Create it on first use if it doesn't exist.

## Naming conventions

```
align-{context-slug}-YYYY-MM-DD.html
align-{context-slug}-YYYY-MM-DD.md
```

- `context-slug` is a kebab-case identifier from the context catalog (see `references/claim-extraction-rhythm.md` for the rhythm catalog, or pick a clean kebab-case slug for new sources).
- If multiple alignment sessions happen on the same day with the same slug, append `-2`, `-3`, etc.: `align-rhythm-2026-03-15.html`, `align-rhythm-2026-03-15-2.html`.
- Filenames are case-sensitive. Use lowercase.

## Manifest format

`rhythm/align-index.md` is a markdown file with three sections:

1. **Header block** — title + one-line description of what the file is.
2. **Manifest table** — one row per session, in chronological order.
3. **(Optional) Patterns + Accuracy Rates sections** — narrative summaries, regenerated by `/retro`. Not authoritative; the manifest table is.

### Manifest table schema

```markdown
| # | Date | Source | Claims | ✅ | ❌ | 🔶 | 🔷 | 🤷 | ➖ | ⬜ | File |
|---|------|--------|--------|---|---|---|---|---|---|---|------|
| 1 | 2026-03-11 AM | /rhythm full — morning brief | 15 | 9 | 1 | 2 | 2 | 0 | 0 | 1 | `align-rhythm-2026-03-11.md` |
```

**Fixed column order, do not reorder:**

| Column | Type | Meaning |
|--------|------|---------|
| `#` | int | Sequential session number, starts at 1, never reused |
| `Date` | string | `YYYY-MM-DD` or `YYYY-MM-DD AM/PM` if multiple sessions per day |
| `Source` | string | Short human-readable description of where the claims came from. Not a folder path. Examples: `"/rhythm full — morning brief"`, `"todo-0515 + horizon + calendar"`, `"v3 Phase 0 Report (Inspect AI benchmark)"` |
| `Claims` | int | Total claim count |
| `✅` | int | Count of correct ratings |
| `❌` | int | Count of wrong ratings |
| `🔶` | int | Count of "almost" ratings |
| `🔷` | int | Count of "needs nuance" ratings |
| `🤷` | int | Count of "can't verify" ratings (typically from `verifiable: false`) |
| `➖` | int | Count of "irrelevant" ratings |
| `⬜` | int | Count of "skipped / unrated" |
| `File` | string | Backtick-quoted filename of the `.md`, no path prefix |

### Counts must sum to total

```
✅ + ❌ + 🔶 + 🔷 + 🤷 + ➖ + ⬜ = Claims
```

This invariant is checked at archive time. If the sum doesn't match `Claims`, the session is malformed — usually a missing rating in the form's export. Don't write a malformed row to the manifest; investigate first.

### What does NOT go in the manifest row

- **Missing items count** — surfaces inside the per-session `.md` under `## Missing — Not Captured`, never in the index. The user can grep across `.md` files when needed.
- **Accuracy percentages** — derived from the manifest, not stored in it. The optional Accuracy Rates section recomputes from the row data.
- **Sub-categorization** (which claims about adidas vs which about Duende) — that lives inside the `.md`, not in the index. `/retro` mines it.

## The pending-corrections queue

When smart-memory MCP isn't available during Phase 2, corrections that would have gone to `remember_facts()` are appended to `rhythm/align-archive/align-corrections-pending.md` as a flat YAML list. The format and drain protocol live in `references/integrations.md`. The file lives in the archive folder so `/retro` can see both successful writes and stuck writes when it audits the loop.

## Archive folder layout (current state)

```
rhythm/align-archive/
├── align-index.md                              # the manifest
├── align-corrections-pending.md                # queue (created on first guarded write)
├── align-rhythm-2026-03-11.html
├── align-rhythm-2026-03-11.md
├── align-rhythm-day-2026-03-11.html
├── align-rhythm-day-2026-03-11.md
├── align-rhythm-2026-03-12.html
├── align-rhythm-2026-03-12.md
... (one .html + one .md pair per session)
```

No subfolders. Flat layout; the manifest is the index.

## What `/retro` expects

`/retro` reads:
1. The manifest table (for session-level metrics).
2. Each individual `.md` file's `## Corrections Required` section (for failure-mode mining).
3. Each `.md`'s `## Missing — Not Captured` section (for pipeline-blind-spot mining).
4. The pending-corrections queue (for "would have written but couldn't" patterns).

It does NOT need to read the `.html` files — those are for human re-inspection only. The convention this plugin follows: the `.html` is consumed in the form-rendering pass; the `.md` is what carries the structured signal downstream.
