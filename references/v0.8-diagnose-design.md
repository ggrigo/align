# `/diagnose` — design doc (v0.8 candidate)

Status: design draft, awaiting review. The naming and the trio framing were locked in conversation 2026-05-26.

## The trio

`/align` is one part of a three-command loop:

| Command | Verb | Time direction | Output |
|---|---|---|---|
| `/align` | **capture** | now | per-claim ratings + reality notes; one pass file per session |
| `/retro` | **forward** | longitudinal | feature candidates / roadmap items the archive suggests |
| `/diagnose` | **backward** | single-session | root cause of the faulty context that produced this session's wrong claims |

`/retro` is product-management-shape: looking at the cumulative archive, finding patterns, proposing what to build next. `/diagnose` is forensic-shape: looking at one session's corrections + the current project state, tracing each wrong claim back to which instructional surface is stale, contradictory, or missing.

`/diagnose` fills the gap between **capture** and **synthesize-across-time**: it runs right after `/align` (or any time the operator wants to revisit a pass file) and answers *"why did this go wrong, and where is the fix?"*

## What it reads

Inputs:

1. **The marked-up pass file** from an `/align` session. The file is the source-of-truth for what the operator marked wrong/almost/nuance, along with their reality-note prose.
2. **The instruction surfaces** in the user's project that could plausibly have caused the misalignment:
   - `CLAUDE.md` (project-root) and any nested `CLAUDE.md` files
   - `skills/*/SKILL.md` (any installed plugin's skill body that ran in the session)
   - `references/` (loaded-on-demand reference docs cited from skill bodies)
   - The producer's emitted output (the original claim array the user rated) if recoverable
   - Recently-modified markdown files in the project root that Claude might be reading
   - **Optionally**: smart-memory entries (the `decisions` collection) if the producer skill has access

What `/diagnose` does **NOT** read:
- Random files in the project (privacy + token cost). Whitelist via the source-scope contract below.
- Files outside the project root (no `~/Documents`, no `/etc`).
- Git history beyond `HEAD~10` (the most recent commits matter; older history doesn't usually).
- Secrets (anything in `.env`, anything matching common secret patterns).

## Source-scope contract

The instruction surfaces `/diagnose` reads are declarative:

```yaml
# Default scope — what /diagnose looks at unless the user narrows it.
diagnose:
  read:
    - CLAUDE.md
    - "**/CLAUDE.md"
    - "skills/*/SKILL.md"
    - "references/**/*.md"
    - "TASKS.md"
  read_with_caveat:
    - smart-memory (decisions collection)
    - recent producer-side output (if recoverable from session)
  ignore:
    - .env
    - node_modules/
    - .git/
    - secrets/
```

The user can narrow at runtime: `/diagnose <pass-file> --scope=CLAUDE.md` reads only the project's CLAUDE.md.

## Confidence-marking

Every per-claim trace ends with a confidence label:

- **high**: single clear source, no contradictions, the trace is verifiable by reading the cited line.
- **medium**: a plausible cause exists but more than one explanation could fit. The trace names the most likely.
- **low / speculative**: multiple plausible causes, no single one dominant; or the trace is inference-heavy. Marked clearly as speculative.

The user reads confidence to prioritize. **High = fix today. Medium = investigate. Low = wait for `/retro` to confirm pattern.**

## Granularity decision

Two granularities; both supported:

1. **Per-claim** (default): each ❌, 🔶, 🔷 in the pass file gets its own trace section. Best for sessions where corrections were heterogeneous.
2. **Per-session** (auto-detected): if multiple claims trace back to the same source, group them in the "Session-level findings" section. Best for sessions where corrections cluster around one root cause.

The skill auto-detects which mode fits by analyzing whether traced-sources overlap. If >50% of ❌s share a single source, surface the session-level finding prominently; otherwise default to per-claim sections.

## Action shape

**v0.8 (this release): report-only.** `/diagnose` writes the markdown report. The user reads, decides, fixes.

**v0.9 candidate: apply-style.** A `/diagnose apply <diagnose-report>` subcommand opens PRs to fix the high-confidence traces (similar to `/retro apply`). Reserved for after the report-only mode has been used in anger and the apply step has clearer semantics.

## Open questions

1. **Smart-memory inclusion**: should `/diagnose` read the smart-memory `decisions` collection (the queue of past corrections)? Pro: a wrong claim might be wrong *because* a past correction wasn't applied to a producer prompt. Con: privacy — smart-memory may have sensitive entries. Recommendation: opt-in via `--include-smart-memory` flag; off by default.

2. **Producer recovery**: many `/align` sessions don't preserve the original producer output. Should `/diagnose` request it explicitly? Recommendation: if the producer is /rhythm-shaped and produced an archivable summary, link it via the pass file's `source` field. Otherwise mark traces as "without producer evidence."

3. **Confidence calibration**: how does the skill decide high vs medium vs low? Initially, heuristics (single-source = high, contradictions = medium, no-source = low). After v0.8 ships, may want to calibrate against user feedback (`/diagnose` outputs themselves become `/align`-able).

4. **Recursion**: `/diagnose`'s own report is an LLM output. Can it be `/align`-ed? Yes — same Claim Adapter Contract. The diagnose report's traces ARE claims about the source code; the user can rate whether each trace is correct. The recursion holds.

5. **Failure mode — no sources found**: if `/diagnose` can't find a source for a wrong claim (the misalignment isn't in any whitelisted surface), should it expand scope? Or report "no source found in scope; consider widening with `--scope=...`"? Recommendation: the latter — keep the scope contract tight; ask the user to opt in to wider reads.

## Risks

- **Confidence inflation**: LLMs tend to confidently identify causes that aren't actually causes. Mitigation: the confidence labels above; explicit prompting in the skill body to favor "speculative" over confident-but-wrong; user-facing language that doesn't read as authoritative.
- **Source-scope creep**: pressure to add more files to the whitelist will accumulate. Resist; the operator can opt-in to wider scope per-run.
- **`/diagnose` becomes `/retro` at single-session scale**: avoid feature drift. Diagnose is backward (cause-tracing); retro is forward (pattern detection). When in doubt about a feature request, ask: *is this looking at one session and asking "why"*? Or *is this looking at many sessions and asking "what next"*?
- **Action shape coupled to report shape**: if v0.9 adds apply-style, the report format must be parseable (the `## Recommendations` table is already structured; keep it that way).

## Implementation sketch

1. `commands/diagnose.md` thin entrypoint → `skills/diagnose/SKILL.md` canonical body.
2. The skill body covers:
   - Argument parsing (`<pass-file>` required; `--scope=...`, `--include-smart-memory` optional).
   - Source-reading (whitelisted file globs; respect `.gitignore`).
   - Per-claim diagnosis prompt (the LLM reasoning step; structured to favor cited evidence).
   - Output formatting (the markdown shape above).
   - Archive: write to `~/.align-archive/diagnose-{slug}-{date}.md`.
3. Manifest schema in `references/archive-format.md` extended with a `diagnose-{slug}` row type (sum-invariant doesn't apply; diagnose reports aren't graded directly though they can be `/align`-ed).
