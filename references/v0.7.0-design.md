# /align v0.7.0 — design

This document scopes the v0.7.0 release of `/align`. It bundles four coordinated changes that ride a shared migration window. Like `references/retro-design.md`, this is a design draft: nothing in it is implemented yet; the document grounds the discussion that produces the implementation PRs.

Status: design draft. Implementation PRs are tracked as separate issues. Each section ends with an *Open questions* note where there's a real choice to make.

## Scope

Four changes for v0.7.0, ordered by dependency:

1. **Multi-skill plugin restructure** (deferred from v0.6.0).
2. **`/retro apply`** — the human-review-gated patch-applying side of the synthesis loop.
3. **"Revisit earlier ratings" affordance** in the form template.
4. **Opt-in smart-memory content filter.**

These bundle for two reasons:

- **Shared migration surface.** (1) changes the file layout. (2) ships under the new layout from day one. (3) and (4) touch user-visible behavior and benefit from one upgrade event rather than three.
- **Coherent narrative.** Each item closes a small loop the project has left open: the restructure enables ergonomic multi-skill growth; `apply` finishes the synthesis-to-fix path; revisit-ratings honors criteria drift; the filter honors the producer-as-trust-boundary discipline already documented in `SECURITY.md`.

Risk in bundling: any one slipping pushes the whole release. Mitigation: (1) and (2) are load-bearing; (3) and (4) ship if ready, defer to v0.7.1 if not.

## (1) Multi-skill plugin restructure

Restructure `/align` from a single-skill plugin to a multi-skill plugin (the recommended shape per `references/retro-design.md` §Where /retro lives Option A):

```
align/
├── .claude-plugin/plugin.json
├── skills/
│   ├── align/SKILL.md
│   ├── align/align-template.html
│   └── retro/SKILL.md
├── references/
├── README.md
├── CONTRIBUTING.md
├── SECURITY.md
└── ...
```

The `commands/` directory either retires or stays as compatibility shims pointing at the new locations. Skill invocation becomes namespaced: `/align:align` and `/align:retro`.

**Pros:** the archive, the manifest schema, and `references/` are shared. One marketplace listing, one install, one version. The Claim Adapter Contract stays inside the same plugin as its first consumer. Standard Claude Code plugin shape going forward.

**Cons:** namespaced skill names read slightly clunky. The restructure touches `/align`'s published entry point; existing users would re-install. The migration is one-time but visible.

### Open questions

- **Compatibility shim or hard cutover?** The plugin.json `commands` declaration can either retire (Claude Code's plugin loader maps `/<plugin>:<skill>` automatically when skills exist) or stay with shim entries pointing at the new locations. The shim form is more conservative; the cutover form is simpler.
- **References shared or per-skill?** Both `/align` and `/retro` read `references/archive-format.md`. Sharing at plugin-root is the right default; the multi-skill structure should not duplicate per-skill.
- **Rollback path.** If the restructure breaks something unanticipated, v0.7.1 should revert by moving files back. Git history makes this reversible. The marketplace listing should treat v0.7.0 as a new minor version; users opting in signal acceptance of the migration.

## (2) `/retro apply`

The human-review-gated patch-applying side of the synthesis loop. `/retro` v0 + v0.1 explicitly defer this to v0.2 of the skill; v0.7.0 of the plugin is the natural release window.

Current state: `/retro` writes pass documents with `⬜ pending` patches. The user manually applies accepted patches by reading the diff and editing the target file. `/retro apply` automates the *acceptance-to-PR* step.

### Input contract

`/retro apply <pass-file-path>` reads a pass document already marked-up by the user:

| Marking | Behavior |
|---------|----------|
| `⬜ pending` | Ignored — user hasn't decided. Skipped without error. |
| `✅ accepted` | Patch is applied as written in the pass document. |
| `❌ rejected` | Patch is skipped; pass-document annotation preserved. |
| `🔶 revised` | Patch is *not* applied; user re-wrote the patch text and a follow-up synthesis tick should redo. |

### Process

1. Validate the pass file: schema check, no `⬜` patches treated as accepted.
2. For each `✅` patch, in order:
   - Read the target file (named in patch metadata).
   - Apply the diff. If diff context doesn't match (file changed since synthesis), abort with a clear message.
   - Validate the patched file by parsing it back (markdown structure, YAML, etc., as applicable to the target type).
3. Group accepted patches by target file. One PR per target file (a CHARTER edit and a SKILL edit go to separate PRs).
4. For each target file with ≥1 accepted patch: open a PR via `gh pr create`. Title: *"Apply /retro patches from pass-N"*. Body: list of accepted patches with source-cluster context.
5. Stop. The user reviews and merges the PRs per the normal /align PR workflow.

### What `/retro apply` does NOT do

- Auto-merge the PRs it opens.
- Modify the pass document (the `✅` / `❌` markings are inputs, never output).
- Cluster across multiple pass documents.
- Apply patches whose target file no longer exists or has moved.

### Recursion

The PRs `/retro apply` opens are LLM outputs. Their PR descriptions are gradable by `/align`; future dogfood passes can grade them, and findings feed back into `/retro` v0.2's spec.

### Open questions

- **Conflict handling.** If a patch's diff context doesn't match the current target file (the file changed since synthesis), the right move is probably to fail cleanly and ask for a re-synth. Silent partial-apply is worse than a clear failure.
- **CI integration?** Probably manual-only for v0.7.0. CI-triggered `/retro apply` is v0.8+ territory.

## (3) "Revisit earlier ratings" affordance

### The phenomenon

[EvalGen (Shankar et al., UIST 2024)](https://arxiv.org/abs/2404.12272) documented *criteria drift*: users iteratively refine their evaluation criteria *while* grading, and even go back to change earlier grades after later items sharpen their criteria. `/align`'s current template is one-pass-then-export. The form has an edit affordance per claim (clicking a rating changes it before export), but the workflow doesn't *surface* that revisiting earlier ratings is part of the process. The taxonomy is the fixed UI; the per-claim reality notes are where criteria evolve — and that evolution should be visible to the user.

### Proposed UI

Minimal change. After the user has graded N claims (suggested threshold: N=5; easy to tune), the form displays a non-intrusive banner above the active claim:

> *Now that you've graded {N}, you might want to revisit your earlier ratings — criteria sharpen as you grade. Click any prior claim to re-rate it.*

The banner is dismissible. The counter resets at 10, 20, 50 so the nudge re-appears at coarse intervals without becoming pestering.

### What this is NOT

- Not a per-claim revisit prompt (intrusive).
- Not a forced two-pass workflow (criteria drift requires *enabling* the behavior, not forcing it).
- Not auto-detection of criteria drift via re-classification consistency (interesting; v0.8+ territory).

### Implementation cost

One paragraph of JS in `align-template.html`; one CSS rule for the banner. Trivial. The hard part is threshold tuning and wording, both of which can be revised based on the first weeks of real usage.

### Open questions

- **Wording in the banner.** The proposed wording above honors criteria drift without making the user feel they were sloppy earlier. Alternatives welcome.
- **Should the banner also appear in `/retro` output?** When `/retro` clusters corrections and proposes patches, the synthesis might benefit from a "your criteria may have drifted; consider re-grading older sessions" note. Likely v0.8+.

## (4) Opt-in smart-memory content filter

### Background

`/align`'s smart-memory integration (per `references/integrations.md` §Smart memory) writes claim text and reality notes to the `decisions` collection without filtering. `SECURITY.md` §What this policy doesn't cover discloses this: *"`/align` writes claim text and your reality note to the `decisions` collection without PII or secret filtering. This is by design — `/align`'s trust boundary is the producer."* Disclosure-only. No defense.

For users whose producers might surface sensitive content (meeting transcripts with attendee names, email bodies, internal discussions), the disclosure-only model means filtering happens at the producer layer or not at all. v0.7.0 introduces an opt-in hook so users can defend without forking the plugin.

### Proposed shape

User (or producer skill) registers a filter function in their config:

```javascript
// User's CLAUDE.md or a producer skill config
align: {
  smartMemoryFilter: (claim, realityNote, producerSlug) => {
    // Return { write: true, claim, realityNote }   → write as-is
    // Return { write: true, claim: scrubbed, realityNote: scrubbed }  → transform
    // Return { write: false }                       → skip the write entirely
  }
}
```

The filter runs once per claim before `/align` calls `remember_facts()`. The filter receives the claim text, the reality note, and the producer slug (so filters can route by producer — strict for meeting-digest, pass-through for code-review, etc.).

Default behavior: no-op (writes everything, matches current behavior — backwards compatible). Users wanting filtering opt in; users who don't get current behavior.

If the filter returns `{ write: false }`, the claim is still recorded in the per-session `.md` for audit; only the smart-memory write is skipped. The archive remains the source of truth; the smart-memory layer is the convenience write.

### What this is NOT

- Not mandatory filtering. The disclosure model still holds — `/align`'s trust boundary is the producer.
- Not a Claude-side filter. No PII/secret detection LLM in the loop; the contract is user-provided logic only.
- Not opt-in by default. Users who care write a filter; users who don't get current behavior.

### Implementation cost

One paragraph of JS in the smart-memory integration code; one config-loading change. Existing tests don't cover smart-memory writes deeply; this is a chance to add a minimal test for filter invocation + the three return shapes.

### Open questions

- **Where does the filter config actually live?** User's `CLAUDE.md`? Per-producer in the producer's own `SKILL.md` frontmatter? Plugin-config file? The cleanest place is probably the user's `CLAUDE.md`, but the per-producer-routing case (different filters per producer) argues for producer-side configuration. Likely both should be supported: a producer can declare a filter; the user can override.
- **Filter chaining?** If both producer-side and user-side filters are registered, which runs first, and can the user's filter override the producer's `{ write: false }`? Probably: producer-side runs first; user-side runs second on the producer-filtered result; user-side can override producer-side `{ write: false }` only by explicit re-declaration.

## Migration plan

`v0.6.0` → `v0.7.0` should be transparent for users who don't engage with the new affordances:

- **Multi-skill restructure.** Marketplace-installed users transparent. `--plugin-dir` users may need to re-clone (one-time). README §Upgrade gets a note.
- **`/retro apply`.** Net new command. Doesn't affect existing `/retro` runs.
- **Revisit-ratings banner.** Net new UI element. Banners dismissible; users who don't want them dismiss once. Default behavior unchanged.
- **Smart-memory filter.** Default no-op = current behavior. Users wanting filtering opt in.

No archive-format changes. No data migration.

---

## What's NOT in v0.7.0

Bundled explicitly so v0.7.0's release notes can be honest about boundaries:

- **`/retro` cluster-across-archives** (v0.8+).
- **Auto-detection of criteria drift** via re-classification consistency (v0.8+).
- **CI integration for `/retro apply`** (v0.8+).
- **Multi-channel messaging / always-on daemon / shell access.** Out of scope by design — `/align` is single-user, local-files, in-your-repo.
- **PII / secret auto-detection in the smart-memory filter.** The filter contract takes user-provided logic; LLM-based detection is out of scope.
