# /align v0.7.0 — design

This document scopes the v0.7.0 release of `/align`. It bundles four coordinated changes that ride a shared migration window. Like `references/retro-design.md`, this is a design that grounds the discussion producing the implementation PRs.

Status: **design + first-cut implementation shipped; validation pending.** All four bundled items have at least one PR landed (see §Implementation sequence below); the remaining work is empirical validation in an interactive Claude session, plus the multi-skill cleanup (#18 steps 3-5). Per-item state is tracked on issues #18-#21. Each section here ends with an *Open questions* note where there's a real choice still to make.

## Scope

Four changes for v0.7.0, ordered by dependency:

1. **Multi-skill plugin restructure** (deferred from v0.6.0).
2. **`/retro apply`** — the human-review-gated patch-applying side of the synthesis loop.
3. **"Revisit earlier ratings" affordance** in the form template.
4. **Opt-in smart-memory content filter.**

These bundle for two reasons:

- **Shared migration surface.** (1) changes the file layout. (2) ships under the new layout from day one. (3) and (4) touch user-visible behavior and benefit from one upgrade event rather than three.
- **Coherent narrative.** Each item closes a small loop the project has left open: the restructure enables ergonomic multi-skill growth; `apply` finishes the synthesis-to-fix path; revisit-ratings honors criteria drift; the filter honors the producer-as-trust-boundary discipline already documented in `SECURITY.md`.

Risk in bundling: any one slipping pushes the whole release. Mitigation: (1) and (2) are load-bearing in *scope* (the others build on the new layout + the apply-side patch contract); (3) and (4) are independent of (2)'s shape and ship if ready, defer to v0.7.1 if not. The actual shipping order didn't follow strict (1) → (2) → (3) → (4) — see §Implementation sequence.

## (1) Multi-skill plugin restructure

Restructure `/align` from a single-skill plugin to a multi-skill plugin (the recommended shape per `references/retro-design.md` §Where /retro lives Option A):

```
align/
├── .claude-plugin/plugin.json
├── skills/
│   ├── align/SKILL.md
│   ├── align/align-template.html
│   └── retro/SKILL.md
├── references/
├── README.md
├── CONTRIBUTING.md
├── SECURITY.md
└── ...
```

The `commands/` directory either retires or stays as compatibility shims pointing at the new locations — that decision lands in step 4 of the migration. During the additive phase (steps 1-2, currently shipped), **both `commands/` and `skills/` are active simultaneously**; the new `skills/` paths take precedence per Claude Code's plugin loader. Skill invocation becomes namespaced: `/align:align` and `/align:retro` (and the legacy `/align` / `/retro` continue to work until step 4 retires them).

**Pros:** the archive, the manifest schema, and `references/` are shared. One marketplace listing, one install, one version. The Claim Adapter Contract stays inside the same plugin as its first consumer. Standard Claude Code plugin shape going forward.

**Cons:** namespaced skill names read slightly clunky. The restructure touches `/align`'s published entry point; existing users would re-install. The migration is one-time but visible.

### Open questions

- **Compatibility shim or hard cutover?** The plugin.json `commands` declaration can either retire (Claude Code's plugin loader maps `/<plugin>:<skill>` automatically when skills exist) or stay with shim entries pointing at the new locations. The shim form is more conservative; the cutover form is simpler.
- **References shared or per-skill?** Both `/align` and `/retro` read `references/archive-format.md`. Sharing at plugin-root is the right default; the multi-skill structure should not duplicate per-skill.
- **Rollback path.** If the restructure breaks something unanticipated, v0.7.1 should revert by moving files back. Git history makes this reversible. The marketplace listing should treat v0.7.0 as a new minor version; users opting in signal acceptance of the migration.

## (2) `/retro apply`

The human-review-gated patch-applying side of the synthesis loop. `/retro` v0 + v0.1 explicitly defer this to v0.2 of the skill; v0.7.0 of the plugin is the natural release window.

Current state: `/retro` writes pass documents with `⬜ pending` patches. The user manually applies accepted patches by reading the diff and editing the target file. `/retro apply` automates the *acceptance-to-PR* step.

### Input contract

`/retro apply <pass-file-path>` reads a pass document already marked-up by the user:

| Marking | Behavior |
|---------|----------|
| `⬜ pending` | Ignored — user hasn't decided. Skipped without error. |
| `✅ accept` | Patch is applied as written in the pass document. |
| `❌ reject` | Patch is skipped. |
| `🔶 revise` | Patch is applied using the user's **revised** diff text written inline in the pass document, not the original. |

This matches the §Review-gate mechanics already specified in `references/retro-design.md` §Stage 2.

### Process

1. Validate the pass file: schema check, no `⬜` patches treated as accepted.
2. For each `✅` or `🔶` patch, in order:
   - Read the target file (named in patch metadata).
   - Apply the diff (the original for `✅`; the user-rewritten text for `🔶`).
   - If diff context doesn't match (file changed since synthesis), abort with a clear message.
   - Validate the patched file by parsing it back (markdown structure, YAML, etc., as applicable to the target type).
3. After all applied: append an **Applied footer** to the pass document recording what landed. Per `retro-design.md` §Stage 2, applied patches are noted so re-runs of `/retro apply` don't double-apply.
4. Stop. Target files are now modified in the working directory. The user handles git / commits / PRs through their own workflow — `/retro apply` does not invoke `git` itself.

### What `/retro apply` does NOT do

- Open PRs or commit to git. Target-file edits live in the working directory; the user (or their agent) decides how to wrap them in commits / PRs / direct-pushes per their workflow.
- Cluster across multiple pass documents.
- Apply patches whose target file no longer exists or has moved.
- Re-apply patches that have already been recorded in the Applied footer (the footer is the deduplication signal).

### Recursion

The patched target files and the appended Applied footer are LLM outputs. They're gradable by `/align`; future dogfood passes can grade them, and findings feed back into `/retro` v0.2's spec.

### Open questions

- **Conflict handling.** If a patch's diff context doesn't match the current target file (the file changed since synthesis), the right move is probably to fail cleanly and ask for a re-synth. Silent partial-apply is worse than a clear failure.
- **CI integration?** Probably manual-only for v0.7.0. CI-triggered `/retro apply` is v0.8+ territory.
- ~~**Dry-run mode?**~~ **Resolved:** `--dry-run` mode shipped in `commands/retro.md` §Apply — Dry-run mode. Read-only preview; no file writes; no pass-document footer modification.
- **Output schema for the Applied footer.** Per-patch line shape, ordering, idempotency rules. Sketched in `retro-design.md` but worth pinning the exact markdown shape before implementation.

## (3) "Revisit earlier ratings" affordance

### The phenomenon

[EvalGen (Shankar et al., UIST 2024)](https://arxiv.org/abs/2404.12272) documented *criteria drift*: users iteratively refine their evaluation criteria *while* grading, and even go back to change earlier grades after later items sharpen their criteria. `/align`'s current template is one-pass-then-export. The form has an edit affordance per claim (clicking a rating changes it before export), but the workflow doesn't *surface* that revisiting earlier ratings is part of the process. The taxonomy is the fixed UI; the per-claim reality notes are where criteria evolve — and that evolution should be visible to the user.

### Proposed UI

Minimal change. After the user has graded N claims (suggested threshold: N=5; easy to tune), the form displays a non-intrusive banner above the active claim:

> *Now that you've graded {N}, you might want to revisit your earlier ratings — criteria sharpen as you grade. Click any prior claim to re-rate it.*

The banner is dismissible. The counter resets at 10, 20, 50 so the nudge re-appears at coarse intervals without becoming pestering.

### What this is NOT

- Not a per-claim revisit prompt (intrusive).
- Not a forced two-pass workflow (criteria drift requires *enabling* the behavior, not forcing it).
- Not auto-detection of criteria drift via re-classification consistency (interesting; v0.8+ territory).

### Implementation cost

Small: PR #28 was ~12 lines of JS + ~5 lines of CSS (plus dual-edit during the migration window — both root `align-template.html` and `skills/align/align-template.html` modified in lockstep). The hard part is threshold tuning and wording, both of which can be revised based on the first weeks of real usage.

### Open questions

- **Wording in the banner.** The proposed wording above honors criteria drift without making the user feel they were sloppy earlier. Alternatives welcome.
- **Should the banner also appear in `/retro` output?** When `/retro` clusters corrections and proposes patches, the synthesis might benefit from a "your criteria may have drifted; consider re-grading older sessions" note. Likely v0.8+.

## (4) Opt-in smart-memory content filter

### Background

`/align`'s smart-memory integration (per `references/integrations.md` §Smart-memory guard) writes claim text and reality notes to the `decisions` collection without filtering. The disclosure-only model means filtering happens at the producer layer or not at all — that was the v0.6.0 state, and the design motivation for v0.7.0's opt-in filter.

Post v0.7.0 implementation: `SECURITY.md` §What this policy doesn't cover now points at both the default (unfiltered) and the opt-in filter; `references/integrations.md` §Smart-memory filter carries the full contract; `SKILL.md` §Smart-memory filter tells Claude how to apply user-declared rules. The disclosure-only → disclosure-plus-defense transition is shipped across PRs #25, #26, #30.

### Proposed shape

**The filter is prose, not code.** Users declare an `<align-smart-memory-filter>` block in their `CLAUDE.md` with free-form natural-language rules; Claude reads them, applies per-correction, and treats them as authoritative.

```markdown
<!-- in user's CLAUDE.md -->
<align-smart-memory-filter>
For each correction about to be written to smart-memory:
- Skip writes from the meeting-digest producer where claim_text mentions attendee personal info.
- For code-review producer: pass through as-is.
- For any producer: if reality note contains a contract amount in € or $, replace the amount with "[REDACTED]" before writing.
- Otherwise: write as-is.
</align-smart-memory-filter>
```

The rules above are illustrative; the real shape is whatever the user writes. The filter contract is fully specified in `references/integrations.md` §Smart-memory filter (shipped in PR #25).

Per-correction outcomes:
- **Write as-is** — payload unchanged.
- **Write transformed** — payload uses scrubbed values from the filter.
- **Skip** — no smart-memory write. The per-session `.md` archive entry still records the original; only the smart-memory layer skips. The archive remains the source of truth.

Filter applies before `remember_facts()` AND before the pending-queue append (so MCP-available and MCP-unavailable paths see consistent filtering).

Default behavior: no filter declared = unfiltered (matches v0.6.0). Users wanting filtering opt in.

### What this is NOT

- Not mandatory filtering. The disclosure model still holds — `/align`'s trust boundary is the producer.
- Not a Claude-side filter. No PII/secret detection LLM in the loop; the contract is user-provided logic only.
- Not opt-in by default. Users who care write a filter; users who don't get current behavior.

### Implementation cost

Prose section in `references/integrations.md` defining the filter contract (shipped in PR #25); cross-reference in `SKILL.md`'s Phase 2 telling Claude to read and apply the filter before each `remember_facts()` call. The smart-memory integration is prose-driven, not JavaScript-driven — Claude executes the contract by reading and interpreting, not by running JS. Future tests would exercise: (a) absent filter = unchanged behavior, (b) filter with skip rule produces a skipped smart-memory write but a recorded archive entry, (c) filter with transform rule produces a transformed smart-memory write and an original archive entry.

### Open questions

- **Where does the filter config actually live?** User's `CLAUDE.md`? Per-producer in the producer's own `SKILL.md` frontmatter? Plugin-config file? The cleanest place is probably the user's `CLAUDE.md`, but the per-producer-routing case (different filters per producer) argues for producer-side configuration. Likely both should be supported: a producer can declare a filter; the user can override.
- **Filter chaining?** If both producer-side and user-side filters are registered, which runs first, and can the user's filter override the producer's `{ write: false }`? Probably: producer-side runs first; user-side runs second on the producer-filtered result; user-side can override producer-side `{ write: false }` only by explicit re-declaration.

## Migration plan

`v0.6.0` → `v0.7.0` should be transparent for users who don't engage with the new affordances:

- **Multi-skill restructure.** Marketplace-installed users transparent. `--plugin-dir` users may need to re-clone (one-time). README §Upgrade gets a note.
- **`/retro apply`.** Net new command. Doesn't affect existing `/retro` runs.
- **Revisit-ratings banner.** Net new UI element. Banners dismissible; users who don't want them dismiss once. Default behavior unchanged.
- **Smart-memory filter.** Default no-op = current behavior. Users wanting filtering opt in.

No archive-format changes. No data migration.

---

## What's NOT in v0.7.0

Bundled explicitly so v0.7.0's release notes can be honest about boundaries:

- **`/retro` cluster-across-archives** (v0.8+).
- **Auto-detection of criteria drift** via re-classification consistency (v0.8+).
- **CI integration for `/retro apply`** (v0.8+).
- **Multi-channel messaging / always-on daemon / shell access.** Out of scope by design — `/align` is single-user, local-files, in-your-repo.
- **PII / secret auto-detection in the smart-memory filter.** The filter contract takes user-provided logic; LLM-based detection is out of scope.

## Implementation sequence

After this design doc lands, the suggested order:

1. **Open tracking issues** in `ggrigo/align` for (1), (2), (3), (4) — one issue per scoped change.
2. **(1) Multi-skill restructure** first. Everything else builds on the new layout.
3. **(2) `/retro apply` and (4) Smart-memory filter** in parallel. Independent code paths; can ship as separate PRs.
4. **(3) Revisit-ratings affordance** last. UI polish; defers to v0.7.1 if the release window is tight.
5. **Release notes + scorecard** when all four (or three + (3) deferred) ship.

1. **Tracking issues opened** — [#18](https://github.com/ggrigo/align/issues/18), [#19](https://github.com/ggrigo/align/issues/19), [#20](https://github.com/ggrigo/align/issues/20), [#21](https://github.com/ggrigo/align/issues/21).

3. **(4) Smart-memory filter** — full 3-PR thread shipped *before* (2) started: PR #25 (contract), #26 (SECURITY), #30 (SKILL.md cross-references).
4. **(3) Revisit-ratings affordance** — first cut: PR #28. Shipped *third* in actual order, not last.

6. **Release notes + scorecard** — pending. Will land after empirical validation closes the open gates.

The order divergence: (4) finished before (2) started, and (3) shipped before (2). Two reasons:

- (3) was bounded UI work (template patch + dual-edit) that didn't depend on (2)'s contract.
- (2) had a spec reconciliation gate (PR #29 fixed contradictions between this design doc's §(2) and `retro-design.md` §Stage 2). The reconciliation took an extra tick before implementation could ship.

The plan's "load-bearing" framing turned out to mean *scope-bearing* (others build on (1)'s layout + (2)'s patch contract), not *sequence-bearing* (must ship first in time).
